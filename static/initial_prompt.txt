4. Web Search Agent (WSA) - the agent will perform web queries to popular search
- Goal
	- The agent receives a user query string and returns a JSON array of pairs: [ [url, content], ... ].
	- The agent must (1) search to obtain a ranked list of URLs, then (2) fetch each URL and extract content.

- Inputs
	- query: string (required)
	- max_results: integer (optional, default 5, range 1–20)
	- language: string (optional, IETF tag like "en", "bg")
	- region: string (optional, e.g. "US", "BG")
	- allow_domains / block_domains: lists (optional)
	- timeout_ms: integer (optional)

- Outputs (STRICT)
	- Output must be valid JSON (no markdown, no extra keys, no commentary).
	- Output must be a JSON array of length <= max_results.
	- Each element must be a 2-item array: [ "<url>", "<content>" ].
	- URL must be absolute and normalized.
	- Content must be UTF-8 text.

- Core Functional Requirements
	1) Search Phase
   		- Use a web search provider/API to retrieve a ranked list of candidate URLs for the query.
   		- De-duplicate URLs (canonicalize; remove tracking params where safe).
   		- Respect allow_domains/block_domains if provided.
   		- Prefer authoritative sources; avoid spam and low-quality pages.
   		- Provide URLs in the same order they will be fetched (highest relevance first).

	2) Fetch Phase
   		- For each URL, fetch the resource over HTTP(S).
   		- Follow redirects up to a safe limit (e.g., 5).
   		- Enforce per-request timeout and overall timeout.
   		- Handle HTTP errors gracefully (skip or return empty content per policy; define policy).

	3) Content Handling
   		- Detect content type using HTTP headers + sniffing fallback.
   		- If content is HTML:
       			- Extract the main readable text only (article/body content).
       			- Remove navigation, sidebars, headers/footers, ads, cookie banners, scripts, styles, and other layout/boilerplate.
       			- Preserve meaningful structure as plain text:
           			- Paragraph breaks
           			- Headings as lines
           			- Lists as bullet lines
           			- Tables as reasonable text (row-per-line) if feasible
       			- Strip all remaining HTML tags and entities.
   		- If content is non-HTML (e.g., PDF, JSON, TXT, XML, CSV, images, binaries):
       			- Return the content unmodified (as text). If binary, specify an exact policy:
           			- Either return base64-encoded bytes as a string, OR
           			- Skip and return empty string with the URL still included (choose one and justify).
       			- Do not attempt summarization or transformation.

	4) Ordering and Completeness
   		- Fetch URLs sequentially or concurrently (define concurrency limit).
   		- Return results in the same order as the selected URLs.
   		- Ensure each returned pair corresponds to exactly one fetched URL.

- Non-Functional Requirements
	- Performance: support concurrent fetches; typical latency target (e.g., <10s for 5 URLs on average network).
	- Reliability: retries with backoff for transient failures; circuit breaking for repeated failures.
	- Security: SSRF protection, block local नेटवर्क ranges, limit redirects, user-agent identification, robots.txt policy (define).
	- Privacy: do not leak user secrets in URLs; strip sensitive query params if needed.
	- Compliance: respect site terms and rate limits; include attribution metadata if required (but DO NOT include it in output JSON).
	- Observability: log query, chosen URLs, status codes, content types, extraction success, timing, errors (without PII).

- Edge Cases to Cover
	- Duplicate URLs, shortened URLs, redirect loops
	- Paywalled pages / consent walls
	- Very large pages (set max bytes)
	- Non-UTF8 encodings (must decode)
	- Mixed content types, incorrect headers
	- Empty/main-text extraction failure (fallback to body text, then to stripped HTML)

- Acceptance Criteria (make these explicit and testable)
	- Given a query, the agent returns a valid JSON array of [url, content] pairs only.
	- For HTML pages, content contains only main readable text with boilerplate removed.
	- For non-HTML resources, content equals the fetched content bytes/text per the chosen “unmodified” policy.
	- Results are ordered and limited by max_results.
	- Agent handles failures without crashing and follows the defined error policy.

- Deliverables
	- Requirements spec (functional + non-functional)
	- API contract (input/output schema)
	- Error handling policy
	- Test plan with at least 10 test cases (HTML article, blog with heavy nav, PDF, JSON, 404, redirect, consent wall, huge file, non-UTF8 page, duplicate links)

	1) A concise requirements document with numbered requirements.
	2) A JSON schema for the output.
	3) A short test plan (bulleted).
